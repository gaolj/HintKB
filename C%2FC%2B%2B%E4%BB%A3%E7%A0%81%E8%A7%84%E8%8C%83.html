1        文件结构

1.1 文件组织

1.1.1           【规则 1-1- 1 】

每个独立模块都要有自己的版本历史说明，用以标明该模块的历史变迁，版本与备份一致，用 txt 文档编写。

版 本 号： v 1.0

完成时间： 2009-12-5

版本特色：完成了读写接口，可正常运行

1.1.2           【规则 1-1- 2 】

构成文件名的每个单词的首字母大写，其余小写，文件名要充分体现本文件所需实现的功能，对应的 cpp 和 h 文件名相同。如：

DesEncrypt.h

DesEncrypt.cpp

1.1.3           【规则 1-1- 3 】

为了防止头文件被重复引用，应当用 ifndef/define/endif 结构产生预处理块。

1.1.4           【规则 1-1- 4 】

用 #include <filename.h> 格式来引用标准库的头文件（编译器将从标准库目录开始搜索）。

1.1.5           【规则 1-1- 5 】

用 #include "filename.h" 格式来引用非标准库的头文件（编译器将从用户的工作目录开始搜索）。

1.1.6           【规则 1-1- 6 】

非模板类的文件组织方式：将类的声明写在头文件中，类中内联成员函数或模板成员函数的定义写在类声明之后头文件之内，类中非内联函数和非模板函数的定义则写在 .cpp 文件中。例如：

MyClass.h

// 版权和版本声明见第五章，此处省略

#ifndef MYCLASS_H

#define MYCLASS_H

 

#include <math.h> // 引用标准库的头文件

#include "myheader.h" // 引用非标准库的头文件

 

class MyClass

{

public :

//......

bool isEmpty(void);

void show(void);

template <typename T>

void setValue(T val);

private :

//......

};

 

//......

 

inline void show(void)

{

//......

}

 

template <typename T>

void setValue(T val)

{

//......

}

 

#endif

 

MyClass.cpp

// 版权和版本声明见第五章 , 此处省略

#include "MyClass.h"

 

//......

 

bool MyClass::isEmpty(void)

{

//......

}

1.1.7           【规则 1-1- 7 】

模板类的文件组织方式：类的声明和及成员函数的定义全部写在 .h 头文件中，类中的成员函数定义写在类声明之后。例如：

MyClass.h

// 版权和版本声明此处省略。

#ifndef MYCLASS_H

#define MYCLASS_H

 

template <typename T>

class MyClass

{ // 类结构声明

public:

T getValue(void);

//......

};

 

template <typename T>

T getValue(void)

{

//......

}

 

#endif

1.2 版权和版权声明

1.2.1           【规则 1-2-1 】

版权和版本的声明位于头文件和定义文件的开头，主要内容有：

1. 版权信息；

2. 文件名称，标识符，摘要；

3. 当前版本号，作者 / 修改者，完成日期。

//~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

// 创建日期： 2009-12-5

// 作    者：袁开国

// 修改日期：

// 修 改 者：

// 修改说明：

// 类 摘 要：完成 Des 加密全部功能

// 详细说明： ……

// 附加说明：说明该类与哪些类有关系等

//~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

2        排版风格

2.1 缩进与对齐

2.1.1           【规则 2-1-1 】

程序块采用缩进风格编写，缩进使用 Tab 。

2.1.2           【规则 2-1-2 】

函数或过程的开始、类、结构、联合、枚举的定义以及循环、判断、分支选择等语句中的代码都要采用缩进风格。例如：


推荐示例：

int main()

{

//.....

}

for ()

{

//....

}

不推荐示例：

for ()

{

//....

}

int main()

{

//....

}


2.1.3           【规则 2-1-3 】

变量的定义通过添加 Tab 键形成对齐，同一类型的变量最好放在一起。例如：


推荐示例：

rect.top    = 0;

rect.bottom = 50;

不推荐示例：

rect.top = 0;

rect.bottom = 50;


2.1.4           【规则 2-1-4 】

switch 语句中所有的 case 需要对齐， break 需要缩进。例如：


推荐示例：

switch(a)

{

case 1:

//......

break;

case 2:

//......

break;

default:

//......

break;

}

不推荐示例：

switch(a) {

case 1:

//......

break;

case 2:

//......

break;

default:

//......

break;

}


2.1.5           【规则 2-1-5 】

if 语句中的成对的 if … else 要对齐。例如：


推荐示例：

if ()

{

//......

}

else

{

//......

}

不推荐示例：

if () {

//......

}else

{

//......

}

 


2.1.6           【规则 2-1-6 】

do … while 语句中的 while 语句在 do 语句结束的后空一格开始书写，无需和 do 对齐。例如：


推荐示例：

do

{

//......

} while ();

不推荐示例：

do{

//......

}

while ();


2.2 空行

2.2.1           【规则 2-2-1 】

在函数，类，结构，枚举等相对独立的程序块声明结束后，必须加空行以示强调（若不加空行在某些编译器中会给出警告），在同一个语句块内，逻辑上相关的语句之间不加空行，其它地方应加空行以示区别。例如：


推荐示例 1 ：

void Function1(...)

{

......

}

// 空行

void Function2(...)

{

......

}

// 空行

void Function3(...)

{

......

}

 

推荐示例 2 ：

while (condition)

{

statement1;

// 空行

if (condition)

{

statement2;

}

else

{

statement3;

}

// 空行

statement4;

}


 

2.2.2           【规则 2-2-2 】

在变量声明结束后必须加空行以示强调。例如：


推荐示例：

int a;

char b;

 

if ()

{

//......

}

不推荐示例：

int a;

char b;

if () {

//......

}

 


2.2.3           【规则 2-2-3 】

将注释与其上面的代码之间用空行隔开。例如：


推荐示例：

#include <iostream>

 

// 以下为主函数

int main()

{

//......

}

 

不推荐示例：

#include <iostream>

/ / 以下为主函数

int main()

{

//......

}


2.2.4           【规则 2-2- 4 】

代码段的每一个独立功能之间加空行，表示逻辑小段。

2.3 空格

2.3.1           【规则 2-3-1 】

" , ”之后要留空格，如 Function(x, y, z) 。如果 " ; “不是一行的结束符号，其后要留空格，如 ：

for(initialization; condition; update) 。

2.3.2           【规则 2-3-2 】

比较操作符 ">" ， "<" 等，赋值操作符 "=" ， "+=" 等，算术操作符 "+" ， "%" 等，逻辑操作符 "&&" ， "||" 等，位域操作符 "^" ， "<<" 等双目操作符前后加空格。例如：


推荐示例：

a = b + c;

a *= b;

if (b < c);

cout << "a";

不推荐示例：

a=b+c;

a*=b;

if(b<c);

cout<<"a";


2.3.3           【规则 2-3-3 】

"!" ， "~" ， "++" ， "--" ， "* ”， "&" （取地址运算符）等单目运算符前后不加空格。例如：


推荐示例：

*p = a;

flag = !isEmpty;

i++;

不推荐示例：

* p = a;

flag = ! isEmpty;

i ++;


2.3.4           【规则 2-3-4 】

"->" ， "." ， "[]" 前后不加空格。例如：


推荐示例：

a = p->get();

a = (*p).get();

a = p[0];

不推荐示例：

a = p -> get();

a = (*p) . get();

a = p [ 0 ];


2.3.5           【规则 2-3-5 】

函数名之后不要留空格，紧跟左括号 "(" ，与关键字区别。例如：


推荐示例：

void getName();

str = a.getName();

不推荐示例：

void getName ();

str = a.getName ();


2.3.6           【规则 2-3-6 】

左括号后面和右括号前面不加空格，多重括号间不必加空格。例如：


推荐示例：

#include <isotream>

if ((a>=b) && (c<=d))

不推荐示例：

#include < isotream >

if ( ( a>=b ) && ( c<=d ) )


2.3.7           【规则 2-3-7 】

关键字的两边都需要留空格，关键字位于行首时关键字前不需要留空格。例如：


推荐示例：

if (a == b)

for ( ; ; );

不推荐示例：

if(a == b)

for( ; ; );


2.3.8           【规则 2-3-8 】

单行注释语句在 "//" 后空一格再开始书写， "//" 用 Tab 键对齐。例如：


推荐示例：

int a = 0;               // ok

int b = 0;               // ok

不推荐示例：

int a = 0;//error

int b = 0;//error


2.3.9           【规则 2-3-9 】

应当将修饰符 * 和＆紧靠变量名，空格留在修饰符后面。例如：


推荐示例：

char *name;

int *x, y; // 此处 y 不被误解为指针

不推荐示例：

char* name;

int* x, y; // 此处 y 会被误解为指针


2.4 风格

2.4.1           【规则 2-4-1 】

if ， for ， do ， while ， case ， switch ， default 等语句自占一行，且各自的执行语句部分无论多少都要加括号 "{}" 。例如：


推荐示例：

for (int i = 0; i < 10; i++)

{

cout << i;

}

if (a == 0)

{

return 0;

}

不推荐示例：

for (int i = 0; i < 10; i++)

cout << i;

if (a == 0)

return 0;

 

 

 


2.4.2           【规则 2-4-2 】

不允许把多个短语句写在一行中，每一行只准书写一条语句。例如：


推荐示例：

point.x = 0;

point.y = 0;

不推荐示例：

point.x = 0; point.y = 0;

point.x = point.y = 0;


2.4.3           【规则 2-4-3 】

将 public 类型的函数写在前面，而将 private 类型的数据写在后面，采用这种版式主张类的设计“以行为为中心”，重点关注的是类应该提供什么样的接口（或服务）。例如：


推荐示例：

class B

{

public:

void Func1(void);

void Func2(void);

private:

int v1;

float v2;

}

不推荐示例：

class B

{

private:

int v1;

float v2;

public:

void Func1(void);

void Func2(void);

}


2.4.4           【规则 2-4-4 】

代码行最大长度宜控制在 80 个字符以内。代码行不要过长，否则眼睛看不过来，也不便于打印。长表达式要在低优先级操作符处拆分成新行，操作符放在新行之首（以便突出操作符）。拆分出的新行要进行适当的缩进，使排版整齐，语句可读。例如：

推荐示例 :

if ((very_longer_variable1 >= very_longer_variable12)

&& (very_longer_variable3 <= very_longer_variable14)

&& (very_longer_variable5 <= very_longer_variable16))

{

dosomething();

}

virtual Matrix MultiplyMatrix (Matrix leftMatrix,

Matrix rightMatrix);

for (very_longer_initialization;

very_longer_condition;

very_longer_update)

{

dosomething();

}

2.4.5           【规则 2-4-5 】

尽可能在定义变量的同时初始化该变量（就近原则）。例如：


推荐示例：

int width = 10;

int height = 10;

int depth = 10;

不推荐示例：

int width

//......

width = 10;


3        命名规范

3.1 一般规范

3.1.1           【规则 3-1-1 】

标识符应当直观且可以拼读，可望文知意，不必进行“解码”，标识符最好采用英文单词或其组合，便于记忆和阅读，切忌使用汉语拼音来命名，程序中的英文单词一般不会太复杂，用词应当准确。例如：


推荐示例：

int currentValue;

void search();

不推荐示例：

int nowValue; // 英文表意不当

void sousuo(); // 不应用拼音描述


3.1.2           【规则 3-1-2 】

标识符的长度应当符合 "min-length && max-information" 原则。

3.1.3           【规则 3-1-3 】

程序中不要出现仅靠大小写区分的相似的标识符。例如：

不推荐示例：

int x, X; // 变量 x 与 X 容易混淆

void foo(int x); // 函数 foo 与 FOO 容易混淆

void FOO(float x);

3.1.4           【规则 3-1-4 】

程序中不要出现标识符完全相同的局部变量和全局变量，尽管两者的作用域不同而不会发生语法错误，但会使人误解。例如：

不推荐示例：

int count;

//......

for (int count = 0; count < 10; count++);

3.1.5 【规则 3-1-5 】

用正确的反义词组命名具有互斥意义的变量或相反动作的函数等。例如：

推荐示例：

int minValue;

int maxValue;

int SetValue(...);

int GetValue(...);

3.1.5           【规则 3-1-6 】

尽量避免名字中出现数字编号，如 Value1 ， Value2 等，除非逻辑上的确需要编号。例如：

不推荐示例：

int value1;

int value2;

void getValue1();

void getValue2();

3.2 类名

3.2.1           【规则 3-2-1 】

开头大写，单词应清晰表达类的用途和功能。

3.2.2           【规则 3-2-2 】

当名称由多个单词构成时，每一个单词的第一个字母必须大写。例如：


推荐示例：

class MyClass;

class Date;

不推荐示例：

class CDate;

class myClass;


3.3 接口名

3.3.1           【规则 3-3-1 】

开头以表示 Interface 的字母 I 为前缀，单词应清晰表达接口的用途和功能。

3.3.2           【规则 3-3-2 】

当名称由多个单词构成时，每个单词的第一个字母必须大写。例如：


推荐示例：

class ISortable;

class IMyInterface;

不推荐示例：

class Sortable;

class ImyInterface;


3.4 函数名

3.4.1           【规则 3-4-1 】

推荐使用动宾结构，函数名应清晰反映函数的功能，用途。

3.4.2           【规则 3-4-2 】

函数名不使用任何前缀。

3.4.3           【规则 3-4-3 】

函数名最长不得超过 30 个字符，应适当采用缩写的形式。

3.4.4           【规则 3-4-4 】

当名称由多个单词构成时，函数名的首字母小写，之后的单词的首字母大写。例如：


推荐示例：

void setValue(...);

string getMessage(...);

不推荐示例：

void SetValue(...);

string messageGet(...);


3.5 变量名

3.5.1           【规则 3-5-1 】

变量的名字应当使用“名词”或者“形容词＋名词”的形式，变量名应清晰反映变量的功能，用途。

3.5.2           3.5.2 【规则 3-5-2 】

当名称由多个单词构成时 , 变量名的首字母小写 , 之后的单词的首字母大写。

3.5.3           3.5.3 【规则 3-5-3 】

变量名最长不得超过 20 个字符。

3.5.4           3.5.4 【规则 3-5-4 】

变量名 i,j,k 等单个字符仅可以用作局部循环变量。例如：


推荐示例：

int age;

string cityName;

不推荐示例：

int iAge;

string CityName;


3.5.5           3.5.5 【规则 3-5-5 】

如果不得已需要全局变量，则使全局变量加前缀 g _ （表示 global ）。

3.5.6           3.5.6 【规则 3-5-6 】

类的数据成员加前缀 m _ （表示 member ），这样可以避免数据成员与成员函数的参数同名。

类的指针成员加前缀 p _ （表示 point ），这样可以避免指针成员与成员函数的参数同名。

3.5.7           3.5.7 【规则 3-5-7 】

静态变量加前缀 s _ （表示 static ）。

3.6 结构体名

3.6.1           【规则 3-6-1 】

开头以表示 struct 的字母 S 为前缀，单词应清晰表达结构体的用途和功能。例如：


推荐示例：

struct SPoint

{

int x;

int y;

};

不推荐示例：

struct Point

{

int x;

int y;

};


3.6.2           【规则 3-6-2 】

当名称由多个单词构成时，每个单词的第一个字母必须大写。

3.7 枚举名

3.7.1           【规则 3-7-1 】

开头以表示 enum 的字母 E 为前缀，单词应清晰表达枚举的用途和功能。

3.7.2           【规则 3-7-2 】

当名称由多个单词构成时，每个单词的第一个字母必须大写。

3.7.3           【规则 3-7-3 】

枚举成员单词的字母全部大写，若名称由多个单词构成时，各单词之间用下划线隔开。例如：


推荐示例：

enum EColor

{

RED;

GREEN;

BLUE;

YELLOW;

}

不推荐示例：

enum Color

{

Red;

Green;

Blue;

Yellow;

}


3.8 宏和常量的命名

3.8.1           【规则 3-8-1 】

尽量使用含义直观的常量来表示那些将在程序中多次出现的数字或字符串。

3.8.2           【规则 3-8-2 】

单词的字母全部大写，各单词之间用下划线隔开。例如：

推荐示例：

#define MAX 100 /* C 语言的宏常量 */

const int MAX_SIZE = 100; // C++ 语言的 const 常量

const float PI = 3.14159; // C++ 语言的 const 常量

4        代码设计

4.1 表达式设计

4.1.1           【规则 4-1-1 】

如果代码行中的运算符比较多，用括号确定表达式的操作顺序，避免使用默认的优先级。例如：


推荐示例：

if ((a | b) && (a & c))

不推荐示例：

if (a | b && a & c)


4.1.2           【规则 4-1-2 】

不要编写太复杂的复合表达式。例如：

不推荐示例：

i = a >= b && c < d && c + f <= g + h ; // 复合表达式过于复杂

4.1.3           【规则 4-1-3 】

不要有多用途的复合表达式。例如：


推荐示例：

a = b + c;

d = a + r;

不推荐示例：

d = (a = b + c) + r ;


4.1.4           【规则 4-1-4 】

不要把程序中的复合表达式与“真正的数学表达式”混淆。例如：

不推荐示例：

if (a < b < c) // a < b < c 是数学表达式而不是程序表达式

4.2 4.2 条件语句设计

4.2.1           【规则 4-2-1 】

不可将布尔变量直接与 TRUE ， FALSE 或者 1 ， 0 进行比较。例如：


推荐示例：

bool flag;

if (flag)               // 表示 flag 为真

if (!flag)               // 表示 flag 为假

 

 

不推荐示例：

bool flag;

if (flag == TRUE)

if (flag == 1 )

if (flag == FALSE)

if (flag == 0)


 

4.2.2           【规则 4-2-2 】

应当将整型变量用 "==" 或 "!=" 直接与 0 比较。例如：


推荐示例：

int value;

if (value == 0)

if (value != 0)

 

不推荐示例：

int value;

if (value)

if (!value)


4.2.3           【规则 4-2-3 】

不可将浮点变量用 "==" 或 "!=" 与任何数字比较，应该设法转化成 ">=" 或 "<=" 形式。例如：


推荐示例：

if((x>=EPSINON)&&(x<=EPSINON))

 

不推荐示例：

float x;

if (x == 0.0) // 隐含错误的比较


4.2.4           【规则 4-2-4 】

应当将指针变量用 "==" 或 "!=" 与 NULL 比较，指针变量的零值是“空”（记为 NULL ），尽管 NULL

的值与 0 相同，但是两者意义不同。例如：


推荐示例：

if (p == NULL)

if (p != NULL)

 

 

不推荐示例：

if (p == 0)

if (p != 0)

if (p)

if (!p)


4.2.5           【规则 4-2-5 】

if 和 else 最好配对使用。例如：


推荐示例：

if (condition)

{

return x;

}

else

{

return y;

}

return (condition ? x : y);

不推荐示例：

if (condition)

return x;

return y;

if (condition)

return x;

else

return y;

 


4.2.6           【规则 4-2-6 】

在使用 switch 语句时，每个 case 语句的结尾不要忘了加 break ，否则将导致多个分支重叠（除非有意使多个分支重叠）。

4.2.7           【规则 4-2-7 】

不要忘记最后那个 default 分支，即使程序真的不需要 default 处理，也应该保留语句 default:break ；这样做并非多此一举，而是为了防止别人误以为你忘了 default 处理。例如：

推荐示例：

switch (variable)

{

case value1 :

//......

break;

case value2 :

//......

break;

//......

default :

break;

}

4.3 循环语句设计

4.3.1           【规则 4-3-1 】

在多重循环中，如果有可能，应当将最长的循环放在最内层，最短的循环放在最外层，以减少 CPU 跨切循环层的次数。例如：


推荐示例：

for (col=0; col<5; col++ )

{

for(row = 0;row < 100;row++)

{

sum = sum + a[row][col];

}

}

不推荐示例：

for (row=0; row<100; row++)

{

for (col = 0;col < 5;col++ )

{

sum = sum + a[row][col];

}

}


4.3.2           【规则 4-3-2 】

如果循环体内存在逻辑判断，并且循环次数很大，宜将逻辑判断移到循环体的外面。例如：

 


高效但不简洁示例 :

if (condition)

{

for (i=0; i<N; i++)

{

DoSomething();

}

}

else

{

for (i=0; i<N; i++)

{

DoOtherthing();

              }

}

低效但简洁示例 :

for (i=0; i<N; i++)

{

if (condition)

{

DoSomething();

}

else

{

DoOtherthing();

}

}

 

 


4.3.3           【规则 4-3-3 】

不可在 for 循环体内修改循环变量，防止 for 循环失去控制。

4.3.4           【规则 4-3-4 】

建议 for 语句的循环控制变量的取值采用“半开半闭区间”写法。例如：


推荐示例：

for (int x=0; x<N; x++)

{

//......

}

不推荐示例：

for (int x=0; x<=N-1; x++)

{

//......

}


4.4 常量设计

4.4.1           【规则 4-4-4 】

尽量使用含义直观的常量来表示那些将在程序中多次出现的数字或字符串。

4.4.2           【规则 4-4-2 】

在 C++ 程序中只使用 const 常量而不使用宏常量，即 const 常量完全取代宏常量。

4.4.3           【规则 4-4-3 】

需要对外公开的常量放在头文件中，不需要对外公开的常量放在定义文件的头部，为便于管理，可以把不同模块的常量集中存放在一个公共的头文件中。

4.4.4           【规则 4-4-4 】

如果某一常量与其它常量密切相关，应在定义中包含这种关系，而不应给出一些孤立的值。例如：

推荐示例：

const float RADIUS = 100;

const float DIAMETER = RADIUS * 2;

4.4.5           【规则 4-4-5 】

类中的常量不能直接被初始化，只能在类构造函数的初始化表中初始化。例如：


推荐示例：

class A

{

A(int size);

const int SIZE ;

};

A::A(int size) : SIZE(size)

{

//......

}

不推荐示例：

class A

{

const int SIZE = 100;

int array[SIZE];

};

 

 

 


4.5 函数设计

4.5.1           【规则 4-5-1 】

参数的书写要完整，不要贪图省事只写参数的类型而省略参数名字，如果函数没有参数，则用 void 填充。例如：


推荐示例：

void s etValue(int width, int height);

float g etValue(void);

不推荐示例：

void SetValue(int, int);

float GetValue();


4.5.2           【规则 4-5-2 】

参数命名要恰当，应有实际意义，顺序要合理，一般地，应将目的参数放在前面，源参数放在后面。例如：


推荐示例：

void StringCopy(char *dest,

char *src);

不推荐示例：

void StringCopy(char *src,

char* dest);


4.5.3           【规则 4-5-3 】

const 修饰函数参数：

1. 对于非内部数据类型的输入参数，应该将“值传递”的方式改为“ const 引用传递”，目的是提高效率。

2. 对于内部数据类型的输入参数，不要将“值传递”的方式改为“ const 引用传递”，否则既达不到提高效率的目的，又降低了函数的可理解性。例如：


推荐示例：

void Func(const A &a);

void Func(int x);

不推荐示例：

void Func(A a);

void Func(const int &x);


4.5.4           【规则 4-5-4 】

const 修饰函数返回值：

1. 如果给以“指针传递”方式的函数返回值加 const 修饰，那么函数返回值（即指针）的内容不能被修改，该返回值只能被赋给加 const 修饰的同类型指针。

2. 如果函数返回值采用“值传递方式”，由于函数会把返回值复制到外部临时的存储单元中，加 const 修饰没有任何价值。

3. 函数返回值采用“引用传递”的场合并不多，这种方式一般只出现在类的赋值函数中，目的是为了实现链式表达。例如：


推荐示例：

const char * g etString(void);

const char *str = g etString();//ok

int g etInt(void); //ok

A g etA(void); //ok

不推荐示例：

const char * GetString(void);

char *str = GetString(); //error

const int GetInt(void); //error

const A GetA(void); //error


4.5.5           【规则 4-5-5 】

避免函数有太多的参数，参数个数尽量控制在 5 个以内。如果参数太多，在使用时容易将参数类型或顺序搞错。

4.5.6           【规则 4-5-6 】

不要将正常值和错误标志混在一起返回，正常值用输出参数获得，而错误标志用 return 语句返回。例如：


推荐示例：

// 若获取成功 , 函数返回真

bool getInt(int &value);

不推荐示例：

// 返回获取的整型值 , 若失败则返回 -1

int getInt() ；


4.5.7           【规则 4-5-7 】

有时候函数原本不需要返回值，但为了增加灵活性如支持链式表达，可以附加返回值。例如：

推荐示例：

// 函数返回值与 strDest 相同 , 但其可作为其他函数的参数 , 增强了灵活性

char *strcpy(char *strDest ， const char *strSrc);

4.5.8           【规则 4-5-8 】

return 语句不可返回指向“栈内存”的“指针”或者“引用”，因为该内存在函数体结束时被自动销毁。例如：

错误示例 :

char* Func(void)

{

char str[] = "hello world"; //str 的内存位于栈上

//......

return str; // 将导致错误

}

4.5.9           【规则 4-5-9 】

如果函数返回值是一个对象，要考虑 return 语句的效率。例如：


推荐示例：

return s tring(s1 + s2);

 

不推荐示例：

String temp(s1 + s2);

return temp;


4.5.10      【规则 4-5-10 】

函数体的规模要小，尽量控制在 100 行代码之内。

4.5.11      【规则 4-5-11 】

尽量避免函数带有“记忆”功能。相同的输入应当产生相同的输出，带有“记忆”功能的函数，其行为可能是不可预测的，因为它的行为可能取决于某种“记忆状态”。这样的函数既不易理解又不利于测试和维护。在 C/C++ 语言中，函数的 static 局部变量是函数的“记忆”存储器。建议尽量少用 static 局部变量，除非必需。

4.5.12      【规则 4-5-12 】

函数的功能要单一，不要设计多用途的函数。

4.5.13      【规则 4-5-13 】

不仅要检查输入参数的有效性，还要检查通过其它途径进入函数体内的变量的有效性，例如全局变量、文件句柄等。

4.5.14      【规则 4-5-14 】

用于出错处理的返回值一定要清楚，让使用者不容易忽视或误解错误情况。

4.5.15      【规则 4-5-15 】

函数参数缺省值只能出现在函数的声明中，而不能出现在定义体中。例如：


推荐示例：

void Foo(int x = 0, int y = 0);

void Foo(int x, int y)

{

//......

}

不推荐示例：

void Foo(int x = 0, int y = 0);

void Foo(int x = 0, int y = 0)

{

//......

}


4.5.16      【规则 4-5-16 】

如果函数有多个参数，参数只能从后向前挨个儿缺省。例如：


推荐示例：

void Foo(int x, int y=0, int z=0);

不推荐示例：

void Foo(int x=0, int y, int z=0);


4.5.17      【规则 4-5-17 】

运算符重载的规则：


运算符

所有的一元运算符

= () [] ->

+= -= /= *= &= |= ~= %= >>= <<=

所有其他运算符

"." 和 C++ 运算符集中没有的符号

规则

建议重载为成员函数

只能重载为成员函数

建议重载为成员函数

建议重载为全局函数

不能被重载


4.5.18      【规则 4-5-18 】

对已经存在的运算符进行重载时，不能改变优先级规则，否则将引起混乱。

4.5.19      【规则 4-5-19 】

inline 关键字应该只与函数定义体放在一起，而不应与函数声明放在一起使用，仅将 inline 放在函数声明前面不起任何作用。例如：


推荐示例：

void Foo(int x, int y);

//......

inline void Foo(int x, int y)

{

//......

}

不推荐示例：

inline void Foo(int x, int y);

//......

void Foo(int x, int y)

{

//......

}


4.5.20      【规则 4-5-20 】

以下情况不宜使用内联：

1. 如果函数体内的代码比较长，使用内联将导致内存消耗代价较高。

2. 如果函数体内出现循环，那么执行函数体内代码的时间要比函数调用的开销大。

3. 函数中存在递归调用，则无法实现内联。

4.5.21      【规则 4-5-21 】

任何不会修改数据成员的函数都应该声明为 const 类型，如果在编写 const 成员函数时，不慎修改了

数据成员 , 或者调用了其它非 const 成员函数 , 编译器将指出错误 , 这无疑会提高程序的健壮性 .

4.6 内存管理

4.6.1           【规则 4-6-1 】

用 malloc 或 new 申请内存之后，应该立即检查指针值是否为 NULL ，防止使用指针值为 NULL 的内存。例如：

推荐示例：

int *p = new int[10];

if (NULL == p)

{

//throw Exception

}

else

{

//......

}

4.6.2           【规则 4-6-2 】

不要忘记为数组和动态内存赋初值，防止将未被初始化的内存作为右值使用。

4.6.3           【规则 4-6-3 】

避免数组或指针的下标越界，特别要当心发生 " 多 1 ”或者 " 少 1" 操作。例如：

不推荐示例：

int *p = new int[10];

//......

int a = p[10];

4.6.4           【规则 4-6-4 】

动态内存的申请与释放必须配对，防止内存泄漏。

4.6.5           【规则 4-6-5 】

用 free 或 delete 释放了内存之后，立即将指针设置为 NULL ，防止产生“野指针”。例如：

推荐示例：

int *p = new int[10];

//......

delete[] p;

p = NULL;

4.7 类设计

4.7.1           【规则 4-7-1 】

在对类的数据成员进行初始化的时候，将 C++ 固有的数据类型的成员变量放在构造函数体内进行初始化，将自定义数据类型的成员变量放在初始化表中进行初始化。例如：

推荐示例 :

class A

{

public:

A(const B &b, int c);

private:

B m _ B;

int m _ C;

}

//......

A::A(const B &b, int c) : m _ B(b)

{

M _ C = c;

}

4.7.2           【规则 4-7-2 】

若类的成员中含有指针变量，则必须重写拷贝构造函数和赋值函数，否则会出现错误。重写赋值函数的四个步骤：

1. 检查自赋值。

2. 用 delete 释放原有的内存资源。

3. 分配新的内存资源。

4. 返回本对象的引用，以支持链式表达。

4.7.3           【规则 4-7-3 】

派生类的构造函数应在其初始化表里调用基类的构造函数。

4.7.4           【规则 4-7-4 】

基类与派生类的析构函数应该为虚（即加 virtual 关键字）。例如：

正确示例 :

#include <iostream.h>

class Base

{

public:

virtual ~Base() { cout<< "~Base" << endl ; }

};

class Derived : public Base

{

public:

virtual ~Derived() { cout<< "~Derived" << endl ; }

};

void main(void)

{

Base *pB = new Derived; // upcast

delete pB;

}

输出结果为：

~Derived

~Base

如果析构函数不为虚 , 那么输出结果为

~Base

4.7.5           【规则 4-7-5 】

在编写派生类的赋值函数时，注意不要忘记对基类的数据成员重新赋值。

正确示例 :

class Base

{

public:

Base& operate =(const Base& other); // 类 Base 的赋值函数

private:

int m _ A, m _ B, m _ C;

};

class Derived : public Base

{

public:

// 类 Derived 的赋值函数

Derived& operate =(const Derived& other);

private:

int m _ A, m _ B, m _ C;

};

Derived& Derived::operate =(const Derived& other)

{

//1. 检查自赋值

if(this == &other)

{

return *this;

              }

//2. 对基类的数据成员重新赋值

Base::operate =(other); // 因为不能直接操作私有数据成员

//3. 对派生类的数据成员赋值

m_ A = other.m _ A;

m _ B = other.m _ B;

m _ C = other.m _ C;

//4. 返回本对象的引用

return *this;

}

4.7.6           【规则 4-7-6 】

严格的继承规则：若在逻辑上 B 是 A 的“一种”，并且 A 的所有功能和属性对 B 而言都有意义，则允许 B 继承 A 的功能和属性。

4.7.7           【规则 4-7-7 】

组合规则：若在逻辑上 A 是 B 的“一部分”（ a part of ），则不允许 B 从 A 派生，而是要用 A 和其它东西组合出 B 。


推荐示例：

class Head

{

public:

//......

private:

Eye m _ Eye;

Nose m _ Nose;

Mouth m _ Mouth;

Ear m _ Ear;

};

不推荐示例：

class Head : public Eye,

public Nose,

public Mouth,

public Ear

{

//......

};

 

 


5        注释规范

5.1 通则

5.1.1           【规则 5-1-1 】

注释尽可能用“ // ”，而少用 /**/ 。

5.1.2           【规则 5-1- 2 】

注释是对代码的“提示”，而不是文档，程序中的注释不可喧宾夺主，注释太多了会让人眼花缭乱，注释的花样要少。

5.1.3           【规则 5-1- 3 】

如果代码本来就是清楚的，则不必加注释，否则多此一举，令人厌烦。例如：

不推荐示例：

i++; // i 加 1, 多余的注释

5.1.4           【规则 5-1- 4 】

边写代码边注释，修改代码同时修改相应的注释，以保证注释与代码的一致性，不再有用的注释要删除。

5.1.5           【规则 5-1- 5 】

注释应当准确，易懂，防止注释有二义性，错误的注释不但无益反而有害。

5.1.6           【规则 5-1- 6 】

尽量避免在注释中使用缩写，特别是不常用缩写。

5.1.7           【规则 5-1- 7 】

注释的位置应与被描述的代码相邻，可以放在代码的上方或右方，不可放在下方。

5.1.8           【规则 5-1- 8 】

当代码比较长，特别是有多重嵌套时，应当在一些段落的结束处加注释，便于阅读。例如：

推荐示例 :

if (...)

{

//...

while (...)

{

//...

} //endwhile

//...

} //endif

6        其他规范

6.1 编译规范

6.1.1           【规则 6-1-1 】

打开编译器的所有告警开关对程序进行编译。

6.1.2           【规则 6-1-2 】

同产品软件（项目组）内，最好使用相同的编辑器，并使用相同的设置选项。

6.1.3           【规则 6-1-3 】

除非实在由于编译器的原因无法消除某些 Warning ，否则坚决杜绝编译期间的 Warning ，执行适当的类型转换以消除类型不匹配 Warning ，但是转换时要小心。

6.2 代码检查

6.2.1           【规则 6-2-1 】

通过代码走读及审查方式对代码进行检查，代码走读主要是对程序的编程风格如注释，命名等以及编程时易出错的内容进行检查，可由开发人员自己或开发人员交叉的方式进行，代码审查主要是对程序实现的功能及程序的稳定性、安全性、可靠性等进行检查及评审，可通过自审、交叉审核或指定部门抽查等方式进行。

6.3 代码优化

6.3.1           【规则 6-3-1 】

不要一味地追求程序的效率，应当在满足正确性，可靠性，健壮性，可读性等质量因素的前提下，设法提高程序的效率。以提高程序的全局效率为主，提高局部效率为辅。

6.3.2           【规则 6-3-2 】

在优化程序的效率时，应当先找出限制效率的“瓶颈”，不要在无关紧要之处优化。先优化数据结构和算法，再优化执行代码。有时候时间效率和空间效率可能对立，此时应当分析那个更重要，作出适当的折衷，例如多花费一些内存来提高性能。

6.3.3           【规则 6-3-3 】

不要追求紧凑的代码，因为紧凑的代码并不能产生高效的机器码。

6.3.4           【规则 6-3-4 】

尽量使用标准库函数，不要“发明”已经存在的库函数。

6.3.5           【规则 6-3-5 】

尽量不要使用与具体硬件或软件环境关系密切的变量。

6.3.6           【规则 6-3-6 】

如果原有的代码质量比较好，尽量复用它，但是不要修补很差劲的代码，应当重新编写。
